**Root Cause Identified:**
The narrative-parser.ts had two issues preventing proper text formatting:

1. **Markdown Italic Stripping Bug**: The regex `/\*([^*]+?)\*/g` for removing italic markdown (`*text*`) was incorrectly matching asterisk bullet points (`* item`). This was removing list markers before the parser could detect them.

2. **Limited Embedded List Support**: The parser only handled text with perfect double-newline paragraph separation. AI responses often contain "embedded lists" where bullet lists appear inline with paragraphs (e.g., "text\n- item1\n- item2\nmore text"). This caused the parser to treat everything as one blob instead of structured sections.

**Fix Implemented:**
1. Updated italic markdown regex to use negative lookbehind: `/(?<!^)(?<!\n)\*([^*\n]+?)\*/g` - this prevents matching asterisks at start of lines (which are bullet markers)
2. Added intelligent embedded list detection and parsing logic:
   - Detects when text has mixed content (some list items, some non-list lines)
   - Processes line-by-line while respecting blank line paragraph breaks
   - Properly separates paragraphs, lists, and metrics even without perfect double-newline separation
   - Falls back to original logic for pure lists or well-formatted text

### Critical Verification Results

#### 1. Does this fix the text formatting issue? YES

The original problem was two-fold:
1. **Markdown Italic Regex Bug**: The regex `/\*([^*]+?)\*/g` was matching asterisk-based bullet markers (`* item`) as italic markers, removing the asterisks before the parser could detect them as list items
2. **Limited Embedded List Support**: The parser only handled text with perfect double-newline separation, causing AI responses with inline lists to be treated as single text blocks

**Fix Validation:**
- Old regex would remove bullet asterisks: `* Item` → ` Item` (broken list detection)
- New regex with negative lookbehind preserves bullets: `* Item` → `* Item` ✓
- Italic text still stripped correctly: `*text*` → `text` ✓
- Embedded list detection now triggers for mixed content (text + lists)
- Line-by-line parsing respects blank line breaks and identifies list items correctly

#### 2. Are the bug fixes correct (regex, embedded lists)? YES

**Regex Analysis:**

Old pattern: `/\*([^*]+?)\*/g`
- Matches ANY asterisk pair: `*text*` OR `* item` (WRONG)
- Cannot distinguish between italic markers and bullet markers

New pattern: `/(?<!^)(?<!\n)\*([^*\n]+?)\*/g`
- `(?<!^)` - Negative lookbehind: not preceded by line start
- `(?<!\n)` - Negative lookbehind: not preceded by newline
- `([^*\n]+?)` - Non-greedy match of non-asterisk, non-newline chars
- `/g` - Global replacement

Result: Matches `*text*` but NOT `* item` because asterisk is either at line start or right after newline ✓

**Embedded List Logic Analysis:**

Detection algorithm:
```
hasEmbeddedList = (listItemCount > 0) AND (listItemCount < lineCount)
```

This correctly identifies:
- Pure lists (all lines are items): FALSE ✓
- Pure text (no items): FALSE ✓
- Mixed content (some items, some text): TRUE ✓

Processing algorithm:
- Splits by newline (preserving structure)
- Tracks whether we're in a paragraph or list
- Blank lines trigger section breaks
- List items identified by `/^[-*•]\s+/` pattern
- Content properly accumulated and sectioned

Verified with test cases - all scenarios handled correctly ✓
